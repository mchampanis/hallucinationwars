# Hallucination Wars — Playable MVP Plan

**Date:** 2026-04-05  
**Status:** Draft — for review

---

## Vision for MVP

A 2-player game where:
- Each player has units with **editable Lua behavior scripts** (sensors in, actuators out)
- A **planning agent** (LLM) can see the game state and issue commands on the player's behalf
- Commands from the player and from the agent go through the **same vocabulary and rules**
- Units can be given **standing roles** that run continuously (e.g. patrol, guard, attack-nearest)
- Units fight, take damage, and die — there is a win condition

This is the minimal slice that demonstrates the full loop: human + LLM co-directing units with scripted behavior, in a live game against an opponent.

## MVP Milestones

### M1 — Expanded Sensors & Actuators

The Lua API needs enough signal for interesting behavior scripts.

**New sensors (added to `self` table):**
```lua
self.health          -- current HP (0–100)
self.max_health      -- max HP
self.team            -- "red" or "blue"
self.nearby_units    -- array of { id, team, x, z, dist, health }
                     --   filtered to within sensor_range (default 20 units)
```

**New actuators:**
```lua
self.fire_at_id      -- unit id to attack this tick (nil = don't fire)
```

**Engine rules:**
- `nearby_units` is populated per-tick by the engine (simple distance scan, O(n²) is fine for 8–16 units)
- `fire_at_id` is consumed by the engine after tick — if the target is in `attack_range` (default 8 units), apply damage
- Sensor/actuator values are defined per unit type (constants on the `Unit` struct); default values work for all current tanks

**Files:** `src/lua-behavior.ts`, `src/units.ts`

---

### M2 — Combat System

Make units fight and die.

**Damage model:**
- Each unit has `attack_damage` (default 10) and `attack_cooldown` (default 1.5 sec)
- When `fire_at_id` resolves a hit: subtract `attack_damage` from target's health, start cooldown
- Health ≤ 0: unit is removed from scene and unit list
- The engine enforces `attack_range` — script cannot fire at out-of-range targets

**Win condition:**
- All units of one team are dead → other team wins
- Show a simple overlay: "Red wins" / "Blue wins" with a Restart button

**Files:** `src/units.ts`, `src/game.ts`, `src/ui.ts`

---

### M3 — Command Vocabulary

A shared, structured way for both player input and the planning agent to issue commands. This is the connective tissue between human, agent, and game state.

**Command types:**
```typescript
type GameCommand =
  | { type: "move";      unit_ids: number[]; x: number; z: number }
  | { type: "attack";    unit_ids: number[]; target_id: number }
  | { type: "stop";      unit_ids: number[] }
  | { type: "set_role";  unit_ids: number[]; role: Role }
  | { type: "set_script";unit_id: number;  script: string }
```

**Command router** (`src/commands.ts`):
- Single function `applyCommand(cmd: GameCommand, units: UnitManager): string | null`
- Returns an error string if invalid (unknown unit, out of range, etc.), null on success
- All player input is refactored to go through this router
- Agent commands go through the same router — no special path

**Files:** `src/commands.ts` (new), `src/input.ts`, `src/units.ts`

---

### M4 — Role System

Standing orders that the engine evaluates each tick against sensor data, producing actuator outputs without requiring script edits.

**Built-in roles:**
```typescript
type Role =
  | { kind: "none" }                                  // script-only, no role override
  | { kind: "attack_nearest" }                        // find nearest enemy, fire_at_id + move toward
  | { kind: "patrol"; waypoints: {x:number,z:number}[] } // cycle waypoints, attack enemies en route
  | { kind: "guard";  cx: number; cz: number; r: number } // hold area, attack intruders, return
```

**How it interacts with Lua:**
- If a unit has a non-`none` role, the engine computes `moveCommand` and `fire_at_id` from the role logic, then calls the Lua tick as normal
- The Lua script still runs — it can override role outputs if it wants, or just use the defaults
- Role is set via `set_role` command (player or agent)
- Role is shown in the selection panel; player can clear it

**Files:** `src/roles.ts` (new), `src/units.ts`, `src/ui.ts`

---

### M5 — Planning Agent (LLM)

An in-game advisor that sees game state and can issue commands.

**Interface:**
- Chat panel (bottom-left, collapsible) — player types prompts to their agent
- Agent sees a **game state snapshot** (JSON): own units (position, health, role, script summary), visible enemy units, pending commands
- Agent responds with prose + optionally a `commands` array (same `GameCommand` types from M3)
- Commands are previewed before execution — player can approve or dismiss
- API key entered in a settings panel (stored in `localStorage`, never sent to our server)

**What the agent can do:**
- Issue any command in the vocabulary (move, attack, stop, set_role, set_script)
- Explain its reasoning in prose
- Suggest script edits — player can click "Apply" to open the script editor pre-filled

**Minimal implementation:**
- Direct Claude API call from the browser using the player's own key
- Single system prompt describing game rules, unit capabilities, command vocabulary
- No token economy yet — just a visible counter showing calls used this session
- No streaming required for MVP — single response

**Files:** `src/agent.ts` (new), `src/ui.ts`, `src/commands.ts`

---

### M6 — Team Assignment

Currently any player can control any unit. Need basic ownership before 2-player testing.

- URL param `?team=red` or `?team=blue` sets the local player's team
- Only that team's units respond to player input (selection, commands)
- Agent also scoped to the player's team
- No networking — both players load the same URL on different devices, game state diverges (that's fine for now, we're testing the loop not sync)

**Files:** `src/game.ts`, `src/input.ts`, `src/agent.ts`

## Milestone Order & Dependencies

```
M1 (sensors/actuators)
  └─ M2 (combat) — needs fire_at_id actuator
       └─ M5 (agent) — more interesting with real combat
M3 (commands)
  ├─ M4 (roles) — set_role command
  └─ M5 (agent) — agent issues commands
M6 (teams) — can be done anytime, needed before 2-player test
```

Suggested order: **M1 → M2 → M3 → M4 → M6 → M5**

Combat first makes the game playable before the agent arrives. The command vocabulary (M3) can be written in parallel with M2 since it's pure TypeScript with no dependencies.

---

## Explicitly Out of Scope for MVP

- Networking / PeerJS (local split-screen or honor system for now)
- Token economy (session call counter only)
- Fog of war
- Resources and building
- DNA fragments / code stealing
- cannon-es physics (kinematic movement is fine)
- Multiple unit types with different stats (all tanks for now, just default stats)
- LLM streaming responses

## Open Design Questions

1. **Agent command approval** — should commands execute immediately or require player tap-to-approve? Immediate is more fun, approval feels safer for testing.
2. **Role vs script priority** — if a unit has both a role and a custom script, which wins? Current proposal: role provides inputs, script runs on top (can override). Is that right?
3. **Sensor range** — should all units have the same range, or should it be part of the unit definition that a coding agent can tune via script?
4. **Attack-move command** — currently in the input system but wired to nothing. Does it map to a role (`attack_nearest` while moving) or stay as a separate command type?
